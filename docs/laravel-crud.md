# CRUD Generator Architecture

This document explains the metadata-driven CRUD system built on the Xcentric IT packages
(`laravel-crud-controller`, `laravel-code-generator`, `code-generator`). It covers both how the
system works today and ŌĆö since the goal is often to build something similar ŌĆö the design decisions
you would reproduce to build your own.

## Overview

What is colloquially called "the CRUD generator" is **not one thing**. It is three packages plus an
external source of truth, split across two distinct phases:

| Layer | Package                               | Phase | Responsibility |
|-------|---------------------------------------|-------|----------------|
| **Runtime** | `xcentric-it/laravel-crud-controller` | Every HTTP request | A single generic controller serves full REST CRUD for *any* model |
| **Build-time orchestration** | `xcentric-it/laravel-code-generator`  | Scaffolding (CLI) | Fetch schema ŌåÆ generate files ŌåÆ diff/migrate DB |
| **Templating engine** | `xcentric-it/code-generator`          | Scaffolding (CLI) | Twig engine that renders an entity schema into PHP source |

The pivot connecting both phases is a **Superadmin metadata store** ŌĆö an external API
(`superadmin_nginx/api/superadmin/`) where entities, their fields, types, relationships, and
validation rules are defined. That schema is the single source of truth. See
`.claude/docs/03-architecture/superadmin.md` for the Superadmin integration.

```
Superadmin API (entity schema)  ŌöĆŌöĆŌöĆŌöĆ single source of truth
   Ōöé
   Ōö£ŌöĆ[build time]ŌåÆ code-generator (Twig) ŌåÆ Model.php, {Model}Request.php, migrations, Routes/api.php
   Ōöé                                              Ōöé
   Ōöé                              each Model embeds $entityMetaData (the schema, inlined as JSON)
   Ōöé
   ŌööŌöĆ[runtime]ŌöĆŌöĆŌöĆŌåÆ LaravelCrudController reads route {model, namespace}
                   ŌåÆ resolves Model / Request / Policy by naming convention
                   ŌåÆ Spatie QueryBuilder (filter / sort / include)
                   ŌåÆ Action (+ before/after events) ŌåÆ BaseResource JSON
```

## Half 1 ŌĆö The Runtime Controller

This is the core of the system, and the part most worth reproducing. There is essentially **one
controller for the entire application**.

### Routes are plain data

Routes are auto-generated and carry the model identity as route action metadata:

```php
// modules/Common/Routes/api.php  (autogenerated ŌĆö "DO NOT EDIT")
Route::group(["prefix" => "common/user"], static function ($router) {
    $router->get('/{id}', [
        'uses'      => LaravelCrudController::class . '@readOne',
        'model'     => 'user',
        'namespace' => 'Modules\Common',
    ]);
    // ... readMore, create, update, delete, mass-*, relation ops
});
```

### Convention-based resolution

On each request, `ModelHelper` turns the `model` + `namespace` route params into fully-qualified
class names:

- **Model** ŌåÆ `Modules\Common\Models\User`
- **Request** ŌåÆ `Modules\Common\Http\Requests\UserRequest` (falls back to base `LaravelCrudRequest` if absent)
- **Policy** ŌåÆ `{Model}Policy` ŌåÆ `App\Policies\DefaultCrudPolicy` ŌåÆ permissive `LaravelCrudPolicy`
- **Controller** ŌåÆ optional `{Model}Controller`; otherwise the base `LaravelCrudController`

### Endpoints

| Method | HTTP | Route | Purpose |
|--------|------|-------|---------|
| `readOne` | GET | `/{id}` | Single record with includes/filters |
| `readMore` | GET | `/` | Paginated list |
| `create` | POST | `/` | Create + nested relations |
| `update` | PUT | `/{id}` | Update + nested relations |
| `delete` | DELETE | `/{id}` | Delete |
| `massCreate` / `massUpdate` / `massCreateOrUpdate` / `massDelete` | ŌĆö | `/mass-*` | Batch operations |
| `addRelation` / `removeRelation` | PUT/DELETE | `/{id}/relation/{field}[/{relId}]` | Create/delete related entity |
| `attachRelation` / `detachRelation` | PUT | `/{id}/attach-relation/...` | Attach/detach existing entity (M2M/HasMany) |

### Per-request lifecycle

```
authorize (Policy)
  ŌåÆ validate (FormRequest rules)
  ŌåÆ build query (Spatie QueryBuilder: filter[eq:email]=, sort=-created_at, include=...)
  ŌåÆ mutate via an Action  (splits scalar vs relation data, syncs relations by type)
  ŌåÆ fire before/after events
  ŌåÆ return BaseResource JSON  (?append=, ?visible= supported)
```

### Two mechanisms that make it truly generic

1. **Relation-aware writes.** On create/update, `EntityRelationsService` splits scalar fields from
   relation fields, saves the scalar data, then syncs each relation by its Eloquent type via a
   `match()` strategy: `BelongsToMany`ŌåÆsync pivot, `HasMany`ŌåÆsave children, `BelongsTo`ŌåÆset FK,
   `MorphTo`ŌåÆmorph, etc. This lets a single POST accept a nested object graph.

2. **~30 dynamic filter strategies.** `LaravelCrudFilter` parses filter names from the query string
   at runtime (`filter[eq:email]=`, `filter[gte:created_at]=`) and resolves a strategy ŌĆö no per-model
   configuration. Available strategies include: `eq, notEq, lt, lte, gt, gte, contains, startsWith,
   endsWith, in, notIn, isNull, isNotNull, has, eqDate, eqDateTime, jsonContains,
   valueBetweenColumns, treeView, treeViewReverse, documentGraph, multiFieldSearch`.

Query features come from `spatie/laravel-query-builder`:
- `?filter[eq:status]=active` ŌĆö filter
- `?sort=-created_at,user.name` ŌĆö sort (incl. relation-field sorts via subqueries)
- `?include=author,comments` ŌĆö eager loads (with optional per-relation filters)
- `?append=full_name` / `?visible=secret` ŌĆö dynamic accessor/visibility on the resource

### What a developer actually writes

Almost nothing. Across the whole app there are ~22 CRUD controllers, most 0ŌĆō30 lines. You only
subclass to add a hook:

```php
class UserController extends LaravelCrudController
{
    protected function onCreate(ActionPayloadInterface $payload): ExecutableActionResponseContract
    {
        $response = parent::onCreate($payload);
        UserCreatedJob::dispatch($payload->getModel()->toArray())->onQueue('auth');
        return $response;
    }

    protected function getAdditionalFilters(): array
    {
        return ['activeWorkPackages' => FilterActiveWorkPackages::class];
    }
}
```

### Extension points

| Need | How |
|------|-----|
| Post-CRUD side effects | Override `onCreate` / `onUpdate` / `onDelete` |
| Custom filters | `getAdditionalFilters()` returning `['name' => FilterClass::class]` (implements Spatie `Filter`) |
| Validation | Create `{Model}Request extends LaravelCrudRequest` with `rules()` / `messages()` |
| Authorization | Create `{Model}Policy` (or edit global `App\Policies\DefaultCrudPolicy`) |
| Decoupled reactions | Listen to `CrudModelBefore/After{Create,Update,Delete,AddRelation,...}` events |
| Custom write logic | Swap the Action via the container (`getCreateAction()` etc.) |

### The model contract

Models extend `CrudModel` (ŌåÆ `AbstractModel` ŌåÆ Eloquent `Model`). The build-time schema survives
into runtime as an inlined `protected string $entityMetaData` JSON string on each model, which drives
API naming, relation config, and the `entity_meta` block appended to responses.

## Half 2 ŌĆö The Scaffolding Pipeline

Driven by the `bizhive:sync-modules` command (in `laravel-code-generator`):

1. **Fetch** module list + `entity-meta` (fields, types, relationships, builtin/custom validation
   rules) from the Superadmin API via `SuperadminDockerClient` (`per_page=999999`).
2. **Skeleton** ŌĆö `ModuleSkeleton` creates the module directory tree and `module.json`.
3. **Entities** ŌĆö `ModuleEntities` parses the schema into `Entity` objects and, per entity, renders:
    - `Models/{Model}.php`
    - `Http/Requests/{Model}Request.php`
4. **Migrations** ŌĆö `Migrations` introspects the live DB (Doctrine DBAL, `bhg_`-prefixed tables) and
   diffs it against the entity definitions using `xcentric-it/migration-mapper`'s `SchemaComparator`,
   emitting timestamped `database/migrations/generated/*.php` files wrapping raw SQL.
5. **Format** ŌĆö runs php-cs-fixer over the generated `Models/` and `Requests/`.

Routes are generated separately (`GenerateRoutes` command) by reflecting over each module's `Models/`
directory; models can opt out with the `#[SkipRouteGenerate]` attribute.

### The templating engine (`code-generator`)

A small Twig-based engine (`twig/twig`, `symfony/string`):

- `Generator` (factory) loads Twig from `src/templates/` and instantiates a strategy.
- `AbstractStrategy::generate()` is a template method: `prepareParameters($entity)` ŌåÆ
  `Twig::render(templateName, params)` ŌåÆ wrap in `GeneratedCode` (code + filename).
- Concrete strategies: `Laravel\Model` (ŌåÆ `model.html.twig`) and `Laravel\Request`
  (ŌåÆ `request.html.twig`).
- `RelationshipType` maps entity association types (`ASSOCIATION_ONE_TO_MANY`, `..._MANY_TO_MANY`,
  `..._MORPH_TO`, ŌĆ”) to Eloquent relation classes.

Generated models carry `$table`, `$hidden`, `$guarded` (always excludes `entity_meta`), `$casts`,
`@property` PHPDoc, `@OA` OpenAPI annotations, relation methods, and the inlined `$entityMetaData`.
Generated requests carry a `rules()` array built from builtin rules (e.g. `unique:table,col,{id},id`)
and custom rules (`new \App\Rules\Foo('{json-config}')`).

## Building Something Similar

The architecture decomposes into **four independent decisions**. Adopt any subset ŌĆö they are not
all-or-nothing.

### 1. Source of truth for the schema
The Xcentric stack uses an external Superadmin API. Cheaper alternatives:
- JSON/YAML file per entity in-repo
- PHP attributes on the model
- DB introspection

This is the **most important choice** ŌĆö every other layer is a projection of it.

### 2. A generic runtime controller  ŌåÉ highest value / lowest effort
Buildable entirely on its own, with hand-written models and no code generation. Recipe:
- Route metadata (`model` + `namespace`) ŌåÆ convention-based resolution of Model/Request/Policy
- Wrap `spatie/laravel-query-builder` for filter/sort/include
- A thin Action layer with before/after events
- A dynamic JSON resource (append/visible)

Shipping *just this* removes ~80% of CRUD boilerplate.

### 3. Relation-sync engine
The strategy-per-relation-type writer for nested payloads. The fiddliest piece (pivots, morphs,
recursion depth). Only build it if your API must accept nested object graphs.

### 4. The code generator
Twig/Blade templates + a strategy per artifact (Model, Request, Migration). Worth it only once the
schema source and templates have stabilized and you regenerate often enough that hand-editing hurts.

### Recommended path
Start with **#2 alone.** Add **#4** once regeneration frequency justifies it. Add **#3** only if
nested writes are a real requirement.
